## 1. 網站定位

`patronum.guru` = WitnessNoir 數位存證產品 **「家庭與人生階段」垂直的對外網站**。

它合併上位 md 裡兩種網站角色：

1. **公開介紹站**——介紹產品、漸進式證據力、家庭情境範本生態（護城河）。
2. **純前端驗證器入口**（上位 md §9）——第三方（法官／保險／對造）不裝 App，上網站丟 `.ots` ＋ 檔案做驗證。

主題把上位 md §13 的內建範本（車禍、施工、點交）替換成**家庭情境範本目錄**，這正是上位 md §1 所稱的護城河：「任何人都能定義並沉澱出一套可信的取證流程」。

**一句話訴求（首頁 Hero 候選）：**
> 為家庭最重要的時刻，留下可驗證、不可竄改的見證。

## 2. 強制紀律（整站文案與結構都必須遵守）

以下三組規則是硬約束，不是建議。任何頁面文案、元件、CTA 都要通過這三關。

### 2.1 法律用詞紀律（上位 md §11，最高優先）

| 禁止 | 改用 |
|------|------|
| ❌「證據保全」當功能名（民訴 §368 是法院程序） | ✅「數位存證」 |
| ❌ 宣稱平台自簽「等同本人親簽」 | ✅「平台電子簽章」（電子簽章法 §6，非政府許可 CA 的數位簽章） |
| ❌ 宣稱自建／自蓋時戳有效 | ✅ 一律強調「外部第三方錨定（OpenTimestamps／Bitcoin）」 |

**正式用詞表**（全站統一）：原件、雜湊值／數位指紋、時間戳記、平台電子簽章、保管鏈（Chain of Custody）、區塊鏈存證、數位存證（證據包）。
**UI／流程動作詞**：歸檔（Archive）、封存（Seal）。

> 對外法律文案最終須律師覆核（上位 md §15）；本站文案先自我遵守上述紀律，標示「非法律意見」。

### 2.2 產品敘事紀律（上位 md §2、§7）

- **漸進式證據力**：付費是「升級」不是「入場」。免費就能留存在性紀錄；要打官司／理賠才付費封存。**禁止**把文案寫成「要付費才能用」。
- **隱私優先**：伺服器全程不碰原圖，只存雜湊／元資料／時間／保管鏈。這是核心賣點，要主動講。
- **流程優先**：護城河是可成長的取證流程範本，不是區塊鏈／簽章本身。行銷重點放在「該拍什麼、怎麼拍齊」。

### 2.3 技術與視覺硬規則（本次新增）

- **單一 CSS 檔**集中所有設定（色彩、字級、間距 token）。元件不得各自寫死數值。
- **字體大小限制**：採用受控的 type scale（固定階梯，數量受限），不散落 magic number。
- **oklch 顏色規則**：所有顏色用 oklch 定義，集中於 CSS 變數；沿用 `design-tokens` 技能產出實際色票（撰寫 CSS 階段載入該技能）。
- **圖表用 Mermaid**（上位個人全域規則）；本站為 Astro 非 MkDocs，換行可用 `<br/>`；Mermaid 色彩用 hex。
- **webroot 參數**：沿用 `allmoneyback.me` 已驗證的 `DEPLOY_TARGET=preview|production` 慣例，買網址後只改一個 env 即上線（見 §6）。

## 3. 「家庭與人生階段」情境範本目錄

對應四大用途：**保險理賠 ／ 法律糾紛 ／ 驗收 ／ 親情存證**。每個情境是一份「該拍哪些拍攝項目 ＋ 要蓋哪些資訊欄位」的清單（對應上位 md §3 範本定義）。

| 情境 | 主要用途 | 範例拍攝項目 |
|------|----------|--------------|
| 新生兒報到 | 親情存證／身分 | 手環、胎記、出生證明、醫院環境 |
| 寶寶成長里程碑 | 親情存證 | 週歲、身高體重、第一次站立 |
| 兒童傷痕與醫療紀錄 | 保險／法律 | 傷口、病歷、校園意外現場 |
| 婚前婚後財產／聘禮點交 | 法律 | 聘禮清單、貴重物、簽收 |
| 購屋租屋點交 | 驗收／法律 | 現況、漏水、瑕疵、水電錶 |
| 交車驗收 | 驗收 | 車身、里程、配件、刮痕 |
| 裝修維修驗收 | 驗收／法律 | 施工前後、隱蔽工程、用料 |
| 長輩照護紀錄 | 法律（長照糾紛） | 傷口、用藥、居家環境 |
| 醫療過程存證 | 法律／保險 | 處置前後、同意書、現場 |
| 遺產遺物清點見證 | 法律 | 物品清單、保險箱、現場 |
| 寵物認養／特徵 | 親情／法律 | 認養文件、特徵、醫療 |
| 居家意外現場 | 保險理賠 | 損壞、現場全景、細節 |

> MVP 先實作其中數個起手範本（數量於實作計畫決定）；其餘列為內容可成長項。

## 5. 視覺與設計 Token 系統

集中於**單一 CSS 檔**（暫定 `src/styles/tokens.css`，由全站 layout 載入）。

### 5.1 顏色（oklch）

- 全部顏色以 oklch 定義為 CSS 變數，禁止元件內寫死色碼。
- 語意層命名（`--color-bg`、`--color-fg`、`--color-accent`、`--color-muted`、`--color-trust`…），而非具體色名。
- 實際色票值於撰寫 CSS 時以 `design-tokens` 技能產出，符合對比與品牌調性（沉穩、可信、見證感）。

### 5.2 字體大小限制（type scale）

- 採固定階梯（暫定 6〜7 級，如 `--fs-xs/sm/base/lg/xl/2xl/3xl`），全站只能引用這些變數。
- 不允許頁面或元件使用階梯外的字級數值。
- 行高、字重一併 token 化。

### 5.3 其他 token

- 間距（spacing scale）、圓角、陰影、容器最大寬度皆 token 化，集中同一檔。

> Mermaid 圖表色彩用 hex（Mermaid 不吃 oklch，依上位全域規則）；其餘網頁 UI 用 oklch。

## 6. 技術骨架與部署（webroot 一鍵切換）

### 6.1 技術選型

- **Astro 靜態站**（`output: 'static'`），pnpm 套件管理——沿用兄弟站 `allmoneyback.me`、`evidencetoday.news` 的成熟慣例。
- 內容（情境範本、FAQ）以 Markdown／MDX 維護，方便成長。
- 圖表：Mermaid。
- 驗證器（`/verify/`）為純前端 JS：驗雜湊一致性、OpenTimestamps 對 Bitcoin、平台電子簽章（對應上位 md §9、§12.1）。MVP 可先放介面骨架與說明，驗證邏輯依後端封存格式落地後接上。

### 6.2 一鍵切換部署（你的「webroot 參數」需求）

直接複用 `allmoneyback.me/astro.config.mjs` 的 `DEPLOY_TARGET` 模式：

- `DEPLOY_TARGET=preview` → `site = https://<owner>.github.io`、`base = /patronum.guru/`、**不出 CNAME** → 買網址前先在 github.io 看草稿。
- `DEPLOY_TARGET=production`（預設）→ `site = https://patronum.guru`、`base = /`、`astro:build:done` 寫出 `dist/CNAME` → 自訂網域上線。
- CNAME 只在 production 由 build hook 產生（**不放** `public/CNAME`，否則 preview 也會被強制導到自訂網域、破壞 github.io 預覽）。

GitHub Actions：push 到 `main` 預設 `production`；`workflow_dispatch` 可手動選 `preview` 看草稿。沿用兄弟站 `.github/workflows/deploy.yml`。

> 換言之：**買 patronum.guru 之前用 preview 看草稿；買到之後，push main 即以 production 上線，零改碼。**
